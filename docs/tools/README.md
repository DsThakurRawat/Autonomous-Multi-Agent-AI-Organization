# Tool Subsystem — Technical Reference

> **Module**: `tools/`
> **Language**: Python 3.12
> **Key Dependencies**: `structlog`, `asyncio`

---

## 1. Overview

The Tool Subsystem provides agents with the ability to interact with the host filesystem, Git repositories, Docker containers, and external services. Every tool is sandboxed, timed, audited, and follows a strict interface contract. Tools are the **only way** agents can produce side effects on the host machine.

### Design Principles
1.  **Unified Interface**: Every tool returns a `ToolResult` with standardized success/failure semantics.
2.  **Timeout Enforcement**: Every tool execution has a configurable timeout (default: 120s).
3.  **Dry-Run Mode**: All tools support a `dry_run` flag that logs the intended action without executing it.
4.  **Audit Trail**: All invocations are logged via `structlog` for post-mortem analysis.
5.  **Safe Subprocess Execution**: All shell commands use `asyncio.create_subprocess_exec` (not `shell=True`) to prevent injection attacks.

## 2. `BaseTool` — Abstract Base Class

**Source**: [`tools/base_tool.py`](../tools/base_tool.py)

All tools inherit from `BaseTool`, which provides:

### 2.1. `ToolResult`

The standardized return type for all tool executions.

| Field | Type | Description |
| :--- | :--- | :--- |
| `success` | `bool` | Whether the tool completed without errors. |
| `output` | `str` | Stdout or primary output (truncated to 5,000 chars for memory safety). |
| `error` | `str \| None` | Stderr or error message on failure. |
| `exit_code` | `int` | Process exit code (0 = success, -1 = timeout/exception). |
| `duration_ms` | `float` | Execution time in milliseconds. |
| `artifacts` | `list[str]` | Paths to any files generated by the tool. |
| `metadata` | `dict` | Additional context (e.g., the command that was run). |

### 2.2. `__call__` — The Execution Wrapper

When a tool is invoked via `await tool(...)`, the `__call__` method:
1.  Logs the start of execution with truncated arguments.
2.  Wraps the `run()` call in `asyncio.wait_for()` with `TIMEOUT_S`.
3.  Catches `TimeoutError` and generic `Exception`, returning a safe `ToolResult`.
4.  Records `duration_ms` on the result.

### 2.3. `_run_subprocess` — Safe Shell Execution

A protected method for running shell commands safely:
-   Uses `asyncio.create_subprocess_exec` (no shell interpretation).
-   Captures stdout/stderr separately.
-   Supports custom environment variables and working directories.
-   In `dry_run` mode, returns a simulated result without executing.

## 3. Tool Inventory

### 3.1. `LocalFileEditTool`

**Source**: [`tools/file_edit_tool.py`](../tools/file_edit_tool.py) · **Name**: `file_edit`

The signature "surgical mastery" tool. Performs exact string replacement in local files rather than full-file rewrites.

**Parameters**:

| Param | Type | Description |
| :--- | :--- | :--- |
| `file_path` | `str` | Relative path from working directory. |
| `old_string` | `str` | The exact string to find (must match including indentation). |
| `new_string` | `str` | The replacement string. |
| `replace_all` | `bool` | If `False` (default), fails when multiple matches exist. |

**Safety Features**:
-   **Uniqueness Check**: If `old_string` has multiple matches and `replace_all=False`, the tool returns an error asking the agent to provide more context.
-   **Atomic Writes**: Writes to a `.tmp` file first, then uses `os.replace()` for an atomic swap. This prevents partial writes on crash.

---

### 3.2. `GitTool`

**Source**: [`tools/git_tool.py`](../tools/git_tool.py) · **Name**: `git_tool`

Manages Git repository operations for the AI-generated project.

**Supported Actions**:

| Action | Description |
| :--- | :--- |
| `init` | Initialize a new Git repo with a Proximus bot identity. |
| `clone` | Clone a remote repository. |
| `add` | Stage files for commit. |
| `commit` | Create a commit with a message. |
| `push` | Push to a remote. Supports `GITHUB_TOKEN` injection for authenticated pushes. |
| `status` | Show short status (`git status --short`). |
| `diff` | Show file diffs (supports `--staged`). |
| `log` | Show recent commit history. |
| `rewind` | Reset to a specific commit hash. **Security**: Validates hash against `^[a-f0-9]{40}$` regex to prevent shell injection. |
| `commit_all` | Convenience: `add .` + `commit` in one call. |

### 3.3. `BrowserTool`

**Source**: [`tools/browser_tool.py`](../tools/browser_tool.py) · **Name**: `browser_tool`

### 3.4. `LinterTool`

**Source**: [`tools/linter_tool.py`](../tools/linter_tool.py) · **Name**: `linter`

**Supported Linters**:
-   **Python**: `ruff check` (lint), `ruff format` (format), `bandit` (security).
-   **JavaScript/TypeScript**: `eslint`.
-   **Go**: `go vet`, `staticcheck`.

**Source**: [`tools/collaboration_tool.py`](../tools/collaboration_tool.py) · **Name**: `collaboration`

Enables inter-agent communication. One agent can post a question or artifact to the shared collaboration board (a JSON file at `.agent_collaboration_board.json`), and another agent can read it.

### 3.7. `SkillsRegistry`

**Source**: [`tools/skills_registry.py`](../tools/skills_registry.py) · **Name**: `skills_registry`

A capability registry that maps agent roles to their available tool sets, enabling dynamic tool resolution at runtime.

## 4. Adding a New Tool

1.  **Create** `tools/my_tool.py`, inheriting from `BaseTool`.
2.  **Set** `NAME`, `DESCRIPTION`, and `TIMEOUT_S` class variables.
3.  **Implement** `async def run(self, **kwargs) -> ToolResult`.
4.  **Register** the tool in `tools/__init__.py`.
5.  **Grant access** via the `SkillsRegistry` for the appropriate agent roles.
